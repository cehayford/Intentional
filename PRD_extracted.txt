1. Executive Summary
The Intentional Spending Tracker is a modern web-based budget management application designed to
transform Nischa's popular Excel-based 50/30/20 budgeting methodology into an interactive, visually engaging
digital experience. This project reimagines traditional spreadsheet budgeting by combining robust backend data
management with cutting-edge 3D visualizations powered by Three.js.
Key Highlights:
•
Foundation: Based on the proven 50/30/20 budgeting rule (50% Needs, 30% Wants, 20% Savings)
•
Technology Stack: NestJS backend, PostgreSQL database, React frontend, Three.js visualizations
•
Core Innovation: Interactive 3D budget visualizations replacing static spreadsheet charts
•
Formula Parity: All 19 original Excel formulas replicated server-side for accuracy
•
Timeline: 11-week development cycle from setup to deployment
2. Product Overview
The Intentional Spending Tracker transforms budget management from a static spreadsheet exercise into an
immersive digital experience. Users can track income, categorize expenses according to the 50/30/20 rule, and
visualize their financial health through interactive 3D representations.
2.1 Problem Statement
Traditional spreadsheet-based budgeting tools present several challenges:
• Limited Engagement: Static charts and tables fail to motivate consistent budget tracking
• Accessibility Barriers: Requires desktop software (Excel/Google Sheets) and spreadsheet literacy
• Data Portability: Difficult to access across devices or share with financial advisors
• Manual Entry: Prone to formula errors and data entry mistakes
• Limited Insights: Basic charts don't reveal spending patterns or trends effectively
2.2 Solution Approach
Our web application addresses these challenges through:
• Cloud-Based Architecture: Access from any device with a web browser
• Automated Calculations: Server-side formula computation ensures accuracy
• 3D Visualizations: Engaging Three.js scenes make budget data compelling and intuitive
• Progressive Enhancement: Works on desktop and mobile with responsive design
• Data Security: PostgreSQL backend with JWT authentication and encryption

7.3 NestJS Module Architecture
The backend is organized into six core modules following NestJS best practices:
AuthModule: Handles registration, login, JWT generation, refresh token rotation
UsersModule: User CRUD operations, profile management, password resets
BudgetsModule: Budget creation, income allocation, monthly budget lifecycle
IncomeModule: Income source management, multi-source support per budget
ExpensesModule: Expense entry, categorization, search/filter capabilities
AnalyticsModule: computeSummary() service, data exports, visualization endpoints

9. API Specifications
9.1 API Design Principles
The REST API follows these design principles:
• RESTful conventions: Resource-based URLs, HTTP verbs for actions
• Versioning: API version in URL path (/api/v1/...)
• JSON responses: All responses in JSON with consistent error format
• JWT authentication: Bearer token in Authorization header
• Pagination: Cursor-based pagination for list endpoints
• Rate limiting: 100 requests/minute per user

10. Frontend and Visualization Requirements
10.1 React Component Architecture
The frontend follows a component-based architecture with clear separation of concerns:
Pages: Login, Register, Dashboard, BudgetDetail, ExpenseList, Analytics
Layout Components: Header, Sidebar, Footer, Navigation
Feature Components: BudgetForm, ExpenseForm, IncomeForm, ExpenseTable
Visualization Components: BudgetGlobe, ParticleFlow, MonthlyBars, SurplusRing
Utility Components: ProtectedRoute, ErrorBoundary, LoadingSpinner
10.2 Three.js Visualization Specifications
Budget Allocation Globe
Description: 3D sphere divided into three colored segments (Needs=green 50%, Wants=blue 30%,
Savings=gold 20%)
Interaction: Hover to highlight segment, click to drill into category details
Performance Target: 60fps on desktop, 30fps on mobile, LOD optimization
Animated Particle Flow
Description: Particles flowing from income source (top) to three category buckets, rate proportional to
allocation
Interaction: Pause/play animation, adjust speed slider
Performance Target: Max 500 particles for smooth animation
Monthly History Bars
Description: 3D bar chart showing 12 months of spending by category, stacked bars
Interaction: Rotate camera, click bar to view month details
Performance Target: 36 bars (3 categories × 12 months), smooth rotation
Surplus/Deficit Torus Ring
Description: Torus ring that rotates and changes color: green (surplus), red (deficit), size proportional to
amount
Interaction: Tooltip shows exact surplus/deficit amount
Performance Target: Simple geometry, animated rotation
10.3 Responsive Design Requirements
The application must adapt to the following breakpoints:
• Desktop (≥1200px): Full dashboard with all visualizations, sidebar navigation
• Tablet (768-1199px): Stacked visualizations, collapsible sidebar
• Mobile (≤767px): Single-column layout, bottom navigation, simplified 3D scenes
Mobile-specific optimizations:
- Reduce particle count to 200 for particle flow
- Use lower-poly geometry for 3D models
- Implement lazy loading for visualization components
- Touch-optimized controls (pinch-zoom, swipe gestures)

13.2 Critical Path Items
The following items are on the critical path and require special attention:
1. computeSummary() Implementation (Week 5): This service must be 100% accurate before building other
features. Recommend test-driven development with spreadsheet comparison suite.
2. Three.js Performance Optimization (Week 8): Early testing on mobile devices to ensure 30fps target. May
require fallback 2D visualizations if performance goals unmet.
3. JWT Refresh Token Rotation (Week 5): Security-critical feature that affects all authenticated endpoints.
Must be implemented early and tested thoroughly.
4. Database Indexing (Week 6): Add indexes on foreign keys and frequently queried columns to meet
performance targets for large datasets.
